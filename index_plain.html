Plain-language overview

I build reliable systems that keep complex software and AI experiments from quietly breaking.

What I do

A lot of modern research and AI work fails in sneaky ways: the code runs, but the result is wrong because something drifted—an input changed, a function signature changed, a file didn’t get saved, or randomness behaved differently. Over time, that wastes weeks and can corrupt conclusions.

My work is about preventing that. I build fail-closed execution systems that check the setup before a run, enforce what’s allowed while it runs, and always produce an auditable record of what happened. If something is off, the system stops early instead of producing misleading output.

If you want the technical version (with the exact terms and framing), use the button at the top: Translate back to engineering.

What that looks like in practice

  • Runs that stop when assumptions are violated (instead of “finishing” with bad results)
  • Reproducible workflows where the same inputs produce the same outputs
  • Clear audit trails: what ran, with what files, settings, and versions
  • Guardrails that catch drift (changed code, changed data, missing files, changed parameters)

The goal: fewer silent failures, faster debugging, and research you can trust months later.
