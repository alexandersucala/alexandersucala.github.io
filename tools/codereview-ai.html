CodeReview AI Live Beta

Built for engineers who are tired of repeating themselves.

Contributing guides, security policies, testing requirements, architecture decisions. The docs exist, but they don't enforce themselves. The same comments show up on PR after PR. The same sections get pointed to over and over. Senior engineers burn time on things that should be automatic. New contributors push code that violates standards nobody told them about. Knowledge is scattered across wikis, markdown files, and Slack threads, and none of it is wired into the actual review process.

CodeReview AI reads your team's documentation and enforces it on every pull request, automatically. Every word, every section, every update. Findings cite the exact document name and line range so the conversation starts at the right place instead of from scratch. The obvious stuff is already flagged before a human reviewer opens the PR. Reviews can focus on architecture and judgment calls, the things that actually need a person.

And when a comment comes up that isn't in the docs yet, update the doc. CodeReview picks it up automatically before the next review. That comment never needs to be made again. The docs become a living system. Not shelf-ware.

How it works.

Setup takes two minutes. Install the GitHub App, and CodeReview auto-detects your repository. It scans the full file tree, identifies documentation files, and classifies every one into review categories: Security, Architecture, Legal, Style, Onboarding. You see live progress as each file is discovered, fetched, and classified. Multi-topic docs get surgically decomposed into gate-specific sections with exact line ranges using a two-model pipeline: an LLM identifies topical boundaries, then content is extracted deterministically by line number. No re-generation, no drift. Confirm what it found and you're live. No config files. No YAML. No re-setup when docs change.

Five review gates.

Every PR goes through five independent review gates. Each gate pulls only the documents relevant to its category and reviews the diff against that subset. Findings are tagged 🔎 doc-backed or 💭 AI suggestion so your team knows exactly what's policy and what's observation. Doc-backed findings cite the specific document, section title, and line range. AI suggestions are clearly marked as the model's opinion, useful but not authoritative.

  • Security API keys, auth patterns, secrets in code, data exposure, injection risks
  • Architecture Design patterns, module boundaries, dependency rules, API contracts
  • Legal Licensing, CLA requirements, copyright headers, redistribution compliance
  • Style Naming conventions, formatting, linting, import order, code standards
  • Onboarding PR process, testing requirements, commit format, contribution workflow

Two-pass hallucination guard.

Every finding goes through a two-pass verification pipeline before it reaches the PR. Pass 1 generates findings by reviewing the diff against your documentation. Pass 2 is a separate verification model that re-reads each finding against the source document and asks: can this actually be proven from what's written? If the answer is no, the finding gets killed. Findings that can't survive verification never reach the PR comment.

This isn't a confidence threshold. It's a second model independently checking the first model's work. Pass 2 tracks removal rates per gate. If a gate's findings are getting killed at a high rate, it flags the pass 1 prompt for tuning. The system monitors its own accuracy and surfaces drift before it becomes a problem.

Docs stay fresh automatically. Someone updates a security policy, merges a new architecture decision, adds a testing requirement. CodeReview detects the change via SHA comparison before the next review and re-ingests the updated content. No stale rules. No re-setup. The knowledge base tracks the repo.

Results.

Results post as a single PR comment within two minutes of the PR opening. Traffic light at the top. Red means fix this before merge, yellow means heads up, green means good to go. Each gate gets its own section with individual status. Every finding includes the violation type, a description, the source document and line range, and the relevant lines in the PR diff. Expandable details keep the comment scannable. Clean, actionable, and designed to be read in under 30 seconds.

Tested against real repos.

The full pipeline has been tested end to end against production repositories including Cal.com, Stripe, Next.js, and Supabase. Not just the discovery and classification. Real PRs were opened against real codebases with messy multi-topic docs, strict contribution policies, monorepo structures, and thousands of files. The system produced accurate, document-backed findings across all five gates and the results were verified manually against the source documentation. The three-step classification pipeline (filename heuristics → LLM content analysis → user confirmation) handled edge cases like READMEs that mix onboarding with security policies, CONTRIBUTING files that embed style rules, and docs directories with hundreds of markdown files.

Currently in beta and free to use. Actively looking for teams to install it, push real PRs against it, break things, and help shape what comes next.

Under the hood.

CodeReview AI is built on a fail-closed architecture. If the system can't prove a finding from your documentation, the finding doesn't ship. If GitHub is unreachable, the review degrades gracefully and posts an error comment rather than failing silently. If a document can't be fetched during the freshness check, the review runs with cached docs because stale docs are better than no review. Every decision path has a fallback, and every fallback logs why it activated.

The discovery module has its own fallback chain: full tree scan via the git/trees API first (single request, fastest), targeted path checks via the contents API if the tree is truncated, and an empty-with-flag return if the API is down entirely. File classification runs a three-pass pipeline: exact filename match at ~80% accuracy, LLM content analysis at ~95% accuracy, and user confirmation at 100%. The decomposer uses LLM section identification followed by deterministic line-range extraction. The AI identifies boundaries, but content is pulled by line number, not regenerated.

Structured logging with JSON output and request IDs runs through the entire pipeline. Every scan, classification, decomposition, review, and verification step is traceable. Background scan workers run in separate threads so the event loop stays free to serve progress polls. The frontend polls every 500ms for live updates during setup.

V2 Roadmap.

Two tiers. Tier 1 is everything above: document-grounded PR reviews with five gates, two-pass verification, and automatic doc freshness tracking. Tier 2 adds full codebase analysis on setup. When you install, CodeReview scores your entire repository across all five gates with a baseline strength rating for each. A historical dashboard tracks how each gate trends over time. Are your security practices improving since last month? Is architectural drift creeping in? Are new contributors following the PR process? Tier 1 tells you what's wrong with this PR. Tier 2 tells you what's happening to your codebase.
